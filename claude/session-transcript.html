04
User prompt
when I click Enter XR, nothing happens

Claude's response

Suspected an uncaught JS error was silently freezing the render loop. A clarifying-question tool call got rejected by the user's client, so the two went back and forth briefly in plain text before the user pasted the actual browser console output (below).

Diagnosed two distinct issues:

1. Harmless / self-recovering — LocomotionSystem's async initLocomotor() hasn't finished by the time the floor entity qualifies; the error is caught internally and it retries successfully a moment later. Confirmed by reading @iwsdk/core/dist/locomotion/locomotion.js.

2. Actual blocker — Chrome defines a native XRWebGLBinding global. IWER (the desktop XR simulator) is supposed to undefine that global during its own async setup, specifically to force three.js onto a compatibility fallback — but three.js's WebXRManager module caches typeof XRWebGLBinding !== 'undefined' once at import time, and IWER's setup loses that race here. Result: three.js constructs a native binding around IWER's emulated session, throws, and the failed teardown throws a second, uncaught error that kills the whole render loop. Confirmed by reading IWER's XRDevice.js, three.js's WebXRManager.js, and @iwsdk/core's xr.js.

Fix: an inline <script> at the very top of index.html's <head> that force-clears window.XRWebGLBinding before any module script evaluates, gated to localhost only so real headsets keep native WebXR Layers support.
